# 06. Архитектурные решения (ADR)

## ADR-001: Использовать React Native CLI без Expo

**Контекст:** тестовое задание прямо требует React Native без Expo.
**Решение:** использовать текущий React Native CLI проект.
**Альтернативы:** Expo, bare Expo.
**Последствия:** больше native setup ответственности, но требование выполнено напрямую.

## ADR-002: Использовать Zustand как state-manager

**Контекст:** state-manager обязателен; проект уже содержит `zustand`.
**Решение:** хранить posts/details/favorites/loading/error в Zustand store.
**Альтернативы:** Redux Toolkit, MobX, React Context.
**Последствия:** меньше boilerplate, проще проверить; важно не смешивать store с API/storage деталями.

## ADR-003: Использовать MMKV для persistence

**Контекст:** пользователь подтвердил MMKV; `react-native-mmkv` уже установлен.
**Решение:** сохранять список, details и favorites через MMKV adapter. MMKV выбран не из-за обязательной потребности в производительности, а потому что зависимость уже есть в проекте и даёт простой синхронный cache.
**Альтернативы:** AsyncStorage — проще для некоторых проверяющих, но потребовал бы замены уже добавленной storage-зависимости.
**Последствия:** высокая скорость и синхронный API; минус — iOS pod/native setup, поэтому README должен отдельно объяснить setup-команду.

## ADR-004: Использовать fetch вместо axios

**Контекст:** API простой, дополнительных HTTP-возможностей не требуется.
**Решение:** использовать встроенный `fetch`.
**Альтернативы:** axios.
**Последствия:** меньше зависимостей; нужно вручную проверять `response.ok`.

## ADR-005: Repository Pattern для data access

**Контекст:** данные приходят из API и cache, нужен one-time fetch.
**Решение:** UI/store обращаются к repositories/services, а не к API напрямую.
**Альтернативы:** вызывать fetch из store или экранов.
**Последствия:** лучше тестируемость и разделение ответственности.

## ADR-006: Сохранять FakerJS-изображения в enriched model

**Контекст:** изображения должны добавляться один раз и не пересоздаваться.
**Решение:** генерировать URL при первом enrichment и сохранять в MMKV вместе с данными.
**Альтернативы:** генерировать в компоненте или selector.
**Последствия:** стабильные изображения после перезапуска; UI проще.

## ADR-007: Избранное хранить отдельно от posts

**Контекст:** favorite state глобальный и должен быстро меняться.
**Решение:** хранить `favoriteIds` отдельно, а posts/details оставлять как cached data.
**Альтернативы:** мутировать каждый post и details с `isFavorite`.
**Последствия:** меньше дублирования; selectors вычисляют visual state.

## ADR-008: Минимальный дизайн без UI-framework

**Контекст:** дизайн произвольный, важнее функциональность и простота проверки.
**Решение:** использовать стандартные React Native компоненты и `StyleSheet` для аккуратной стилизации.
**Альтернативы:** UI kit.
**Последствия:** меньше зависимостей и рисков.
