Ты работаешь как AI-агент над React Native CLI приложением для тестового задания.

Текущая задача — привести приложение к строгому, аккуратному, современному и максимально близкому к нативному виду для каждой платформы: iOS и Android.

Важно:
- Не переписывай бизнес-логику без необходимости.
- Не ломай текущую архитектуру.
- Не меняй data layer, repositories, persistence, store и navigation-логику, если это не требуется для UI.
- Не добавляй новые зависимости без явной необходимости.
- Если хочешь использовать новую зависимость для UI, сначала зафиксируй рекомендацию в документации и объясни, зачем она нужна.
- Приложение должно остаться React Native CLI, не Expo.
- TypeScript-типизация должна сохраниться.
- UI должен выглядеть строго, чисто, профессионально и близко к нативным приложениям.

Цель:
Сделать визуальный слой приложения качественным:
- аккуратные отступы;
- читаемая типографика;
- приятные карточки постов;
- понятное визуальное выделение избранного;
- нативные press states;
- корректная работа Safe Area;
- аккуратные loading/error/empty states;
- адаптация под iOS и Android;
- поддержка светлой и тёмной темы, если это можно сделать без чрезмерного усложнения;
- хорошая доступность;
- стабильное поведение на разных размерах экранов.

Требования к визуальному стилю:

Общий стиль:
- строгий;
- минималистичный;
- аккуратный;
- без кислотных цветов;
- без перегруженных теней;
- без чрезмерной декоративности;
- похожий на production-приложение;
- с хорошей читаемостью.

Цвета:
- используй нейтральную палитру;
- основной фон — спокойный;
- карточки должны визуально отделяться от фона;
- избранное состояние должно быть заметным, но не кричащим;
- error-состояние должно быть понятным;
- disabled/loading-состояния должны быть визуально отличимы.

Типографика:
- используй системные шрифты;
- не подключай кастомные шрифты;
- заголовки должны быть выразительными, но не огромными;
- body-текст должен быть читаемым;
- secondary-текст должен быть визуально слабее;
- учитывай различия iOS и Android.

Платформенная адаптация:

Для iOS:
- использовать более мягкие скругления;
- аккуратные тени через shadowColor/shadowOpacity/shadowRadius/shadowOffset;
- Pressable feedback через opacity/scale, если это уместно;
- визуально лёгкие карточки;
- кнопки ближе к iOS-паттернам;
- навигационный header должен выглядеть естественно для iOS;
- учитывать Safe Area.

Для Android:
- использовать elevation вместо iOS-теней;
- использовать android_ripple для Pressable, где это уместно;
- чуть более material-like карточки;
- понятный touch feedback;
- header должен выглядеть естественно для Android;
- учитывать StatusBar и Safe Area.

Компоненты:

- добавить accessibilityRole="button";
- добавить accessibilityState selected или checked, если подходит;

Дизайн-токены:

Создай или улучши отдельный файл с токенами, например:
- `src/shared/ui/theme/tokens.ts`
или другое подходящее место по текущей архитектуре.

Токены должны включать:
- spacing;
- radius;
- typography;
- colors;
- shadows;
- platform-specific values.

Пример направления, не обязательно копировать буквально:
- spacing: xs, sm, md, lg, xl;
- radius: sm, md, lg, pill;
- typography: title, subtitle, body, caption;
- colors: background, surface, textPrimary, textSecondary, border, accent, favoriteBackground, error;
- shadows: cardShadow для iOS и elevation для Android.

Архитектурные требования:
- Стили не должны быть хаотично разбросаны.
- Повторяемые значения вынести в tokens/theme.
- UI-компоненты не должны знать о persistence.
- UI-компоненты не должны напрямую выполнять сетевые запросы.
- UI-компоненты должны получать данные из store/selectors/hooks.
- Не смешивай data layer и presentation layer.
- Не добавляй сложный дизайн-системный слой, если он не нужен.
- Соблюдай KISS и YAGNI.

Accessibility:
- Добавь accessibilityRole для кликабельных элементов.
- Добавь accessibilityLabel для элементов списка и кнопки избранного.
- Добавь accessibilityState для избранного состояния, если применимо.
- Убедись, что touch target достаточно большой.
- Не используй только цвет для обозначения избранного; должен быть ещё текстовый или иконоподобный индикатор.

Performance:
- FlatList должен быть оптимизирован.
- Не создавай inline-функции и inline-объекты там, где это может ухудшить производительность без причины.
- Используй memo для PostListItem, если это оправдано.
- Используй keyExtractor.
- Используй useCallback для renderItem/navigation handlers, если это уместно.
- Не пересортировывай список в render без memoized selector.
- Не генерируй FakerJS-данные в UI.
- Не пересоздавай изображения при каждом рендере.
